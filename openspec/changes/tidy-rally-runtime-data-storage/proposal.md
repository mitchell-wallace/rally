## Why

Rally's `.rally/` directory has accreted a flat pile of machine-churned files (`tries.jsonl` at ~1.8MB, a 120KB `current_task.md`, `hook-audit.jsonl`, legacy `batches/` and `relays/` dirs if present) alongside the user-facing `progress.yaml`, which has grown messy and inconsistent with the JSONL-everywhere convention. The `store` spec claims these JSONL files are git-tracked "source of truth", but the current `.rally/.gitignore` template only ignores `current_task.md`, `relays/`, and `run-state.json` — the JSONL files are excluded from git only by user action, not by rally. In practice only `config.toml`, `agents/`, and `README.md` travel via git, so run history never moves between containers and the durability promise is hollow. We want a clean top-level layout, a durable centralised log store (Sentry) instead of relying on bloated git-tracked JSONL, and laps bundled as a first-class companion to rally.

## What Changes

- **BREAKING**: Move all machine-churned data into a new gitignored `.rally/state/` subfolder: `tries.jsonl`, `relays.jsonl`, `agent_status.jsonl`, `messages.jsonl`, `hook-audit.jsonl`, `run-state.json`, `current_task.md`. These are no longer git-tracked; durability shifts to Sentry + local retention.
- **BREAKING**: Replace `.rally/progress.yaml` with an append-only `.rally/summary.jsonl` — the sole top-level data file and the only tracked run-history artifact. Same per-run/handoff record shape, one JSON line each.
- Reduce tracked `.rally/` contents to: `config.toml`, `agents/`, `README.md`, `summary.jsonl`. The new `.rally/.gitignore` is just `state/`.
- Add a one-time, idempotent migration in `runInit`: move existing flat files into `state/` and remove legacy `batches/` and `relays/` log dir (if present — these are legacy artifacts that may not exist in all repos). Existing `progress.yaml` is left as-is; new writes go to `summary.jsonl` only.
- Add an opt-in Sentry telemetry sink (default off; DSN via `config.toml [telemetry]` + `SENTRY_DSN` env; `RALLY_TELEMETRY=0` kill switch). Emits tries as structured logs, relay/run/try as a trace hierarchy, and genuine failures as Issues, tagged with `relay_id`/`run_id`/`try_id`/`role`/`runner`/`repo`/`lap_id`. Issues are reserved for operator-worthy failures (infra-class failures, relay stall with all agents frozen, panic, no-finalize, `laps done`-as-text, lap-integrity violations); ordinary agent-class retries stay spans/logs to avoid alert noise. Route fallback (rotating to a backup runner) is logged as a common recovery event, not an Issue. Each try log also carries the assembled-prompt size and a per-source breakdown so runaway prompt growth is caught (the enforcement-side budget lives in `harden-relay-run-lifecycle`). Flushes on CLI exit; a `before_send` scrubber never ships `current_task.md` or full transcripts.
- Persist the full ordered commit list per try (not just one final hash) so causal chains across tries survive in the try record.
- Bundle the `laps` binary alongside `rally` in `install.sh`, extend the existing `rally update` command to also upgrade laps, and add a startup minimum-laps-version check. `.laps/` stays a separate top-level dir (laps never reads `.rally/`).
- Add a regression test asserting rally never writes a `.laps/.gitignore` (the stray hand-committed `.laps/.gitignore` was already removed and `laps.json` is already tracked as of commit `33d288e`).

## Capabilities

### New Capabilities
- `run-summary`: the append-only `summary.jsonl` digest of finalized runs and handoffs that replaces `progress.yaml` as the sole tracked top-level data file.
- `telemetry`: opt-in Sentry sink emitting structured try logs (with assembled-prompt size + per-source breakdown), relay/run/try traces, and operator-worthy failure Issues (infra failures and relay stalls, not agent-class retries), with config, scrubbing, and CLI-exit flushing.
- `tooling-distribution`: bundling laps alongside rally (install + extending `rally update` + min-version check) and adding a regression test that rally never writes a `.laps/.gitignore`.

### Modified Capabilities
- `store`: JSONL records relocate to gitignored `.rally/state/` and are no longer git-tracked (**BREAKING**); git-commit windowing is removed — append-only log files (`tries.jsonl`, `relays.jsonl`) are never pruned, while read-oriented state files (`agent_status.jsonl`, `messages.jsonl`) retain windowing with more conservative limits. Adds the one-time migration requirement and a per-try commit-history field (full ordered commit list, not a single hash).

- **New deps**: `github.com/getsentry/sentry-go`.
- **New config**: `[telemetry] sentry_dsn` in `config.toml`; `SENTRY_DSN` / `RALLY_TELEMETRY` env vars.
- **Distribution**: `install.sh` fetches/installs laps from https://github.com/mitchell-wallace/laps; existing `rally update` extended to upgrade laps too.

The current `.rally/.gitignore` template only ignores `current_task.md`, `relays/`, and `run-state.json` — the JSONL files were never explicitly gitignored by rally, only excluded by user action.
- **Coordination with `harden-relay-run-lifecycle`** (already landed): that change altered `agent_status.jsonl` freeze semantics (decay to probation + `--new` reset, per-harness-model `model` field, 500-event window), defined failure classification (infra/agent/incomplete), and added `laps_attempted` to the try record. This change relocates `agent_status.jsonl` to `state/` (path-agnostic, no semantic conflict), preserves its windowing/semantics rather than re-specifying them, and consumes the classification for Issue criteria. The per-try commit list (`CommitHistory []string`) is added alongside the existing `laps_attempted` field — no fork of the try record.
- **Not changed**: `dataDir` verbose logs (`~/.local/share/rally/`), laps' own `.laps/` location and standalone usability.
