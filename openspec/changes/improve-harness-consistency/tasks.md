## 4. Codex silent-exit enrichment (session-log fallback)

The `runCodexCommand` merge-stderr pattern (`internal/agent/codex.go:223`,
`cmd.StdoutPipe()` then `cmd.Stderr = cmd.Stdout` before `cmd.Start()`) is
the standard Go merge idiom (shared with `runLoggedCommand` in
`internal/agent/log.go:44`) and is NOT a race — no refactor of stderr
capture is needed for this change. The real silent-exit-1 problem is that
codex writes nothing to *either* stream before dying; the session-log
fallback below is the authoritative safety net.

- [ ] 4.1 Add a codex session-log locator that scans `$CODEX_HOME/sessions/YYYY/MM/DD/rollout-*.jsonl` (default `~/.codex/sessions/` when `CODEX_HOME` is unset) for a file whose first-line `session_meta.cwd` matches `opts.WorkspaceDir` and whose `session_meta.timestamp` is within the try window
- [ ] 4.2 Read only the first line (`session_meta`) for top-level scalars (`cwd`, `git.commit_hash`, `git.branch`, `model_provider`, `cli_version`); explicitly skip `base_instructions`, `token_count`, `response_item`, and `turn_context` payloads. NOTE: the resolved `model` is NOT in `session_meta` (only `model_provider` is); `ResolvedModel` comes from the executor's own `model` local (task 2.2), not the session log
- [ ] 4.3 Read the last `event_msg` line of any subtype; use its `payload.type` (e.g. `task_started`, `task_complete`, `turn_aborted`) as the diagnostic message
- [ ] 4.4 In `CodexExecutor.Execute` (`internal/agent/codex.go:195-201`): when the in-band buffer has no parser-matchable signal and a session-log file matched, populate `FailureEvidence` with `Source = "codex_session_log"`, `Message` from the last event's subtype, and a 256-rune-bounded `RawSignal`
- [ ] 4.5 When no session-log file matches, populate `FailureEvidence` AT THE EXECUTOR LEVEL with `Category = CategoryHarnessLaunch`, `Source = "codex_no_session_log"`, and `Message = "codex launched but wrote no session log"`. This lets `ClassifyError` Priority 1 (typed executor Evidence) classify the failure directly as `harness_launch`. The existing `CategoryHarnessLaunch` mapping applies as-is: `StrategyFreshRestart` (retry within budget with a fresh session) + `FailureInfra` (freeze pressure after 2+ infra failures, which is the right "up to a limit" cap for a harness that keeps failing to launch). The intent is NOT to skip retrying — it is to label the failure correctly and surface the `codex_no_session_log` repro marker so the launch issue can be reproduced and fixed, instead of burning retries as an uncategorised `agent_error`. No runner-side classification change is needed for this case (the prior draft's cross-reference to "task 5" was wrong; task 5 is the try-budget-exhaustion path, unrelated to harness_launch)
- [ ] 4.6 Tests: (a) codex exit-1 with a matching session log → `Source = "codex_session_log"` and expected `Message` from the last event subtype; (b) codex exit-1 with no matching session log → executor returns `Evidence.Category = CategoryHarnessLaunch`, `Source = "codex_no_session_log"`, and `ClassifyError` Priority 1 resolves the decision as `harness_launch` / `StrategyFreshRestart` / `FailureInfra` (NOT StrategyRotate / FailureAgent — those would only come from CategoryAuthOrProxy or CategoryInvalidModel, which are semantically wrong for a launch failure the runner should keep retrying); (c) `base_instructions`, `token_count`, and `turn_context` payloads never appear in `RawSignal`; (d) missing/unreadable session log directory is not an error (falls through to existing `safe_exec_error` path)

the run-budget case by clearing `failureCategory`, setting
