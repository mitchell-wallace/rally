## 4. Codex silent-exit enrichment (session-log fallback)

The `runCodexCommand` merge-stderr pattern (`internal/agent/codex.go:223`,
`cmd.StdoutPipe()` then `cmd.Stderr = cmd.Stdout` before `cmd.Start()`) is
the standard Go merge idiom (shared with `runLoggedCommand` in
`internal/agent/log.go:44`) and is NOT a race — no refactor of stderr
capture is needed for this change. The real silent-exit-1 problem is that
codex writes nothing to *either* stream before dying; the session-log
fallback below is the authoritative safety net.

- [x] 4.1 Add a codex session-log locator that scans `$CODEX_HOME/sessions/YYYY/MM/DD/rollout-*.jsonl` (default `~/.codex/sessions/` when `CODEX_HOME` is unset) for a file whose first-line `session_meta.cwd` matches `opts.WorkspaceDir` and whose `session_meta.timestamp` is within the try window
- [x] 4.2 Read only the first line (`session_meta`) for top-level scalars (`cwd`, `git.commit_hash`, `git.branch`, `model_provider`, `cli_version`); explicitly skip `base_instructions`, `token_count`, `response_item`, and `turn_context` payloads. NOTE: the resolved `model` is NOT in `session_meta` (only `model_provider` is); `ResolvedModel` comes from the executor's own `model` local (task 2.2), not the session log
- [x] 4.3 Read the last `event_msg` line of any subtype; use its `payload.type` (e.g. `task_started`, `task_complete`, `turn_aborted`) as the diagnostic message
- [x] 4.4 In `CodexExecutor.Execute` (`internal/agent/codex.go:195-201`): when the in-band buffer has no parser-matchable signal and a session-log file matched, populate `FailureEvidence` with `Source = "codex_session_log"`, `Message` from the last event's subtype, and a 256-rune-bounded `RawSignal`
- [x] 4.5 When no session-log file matches, populate `FailureEvidence` AT THE EXECUTOR LEVEL with `Category = CategoryHarnessLaunch`, `Source = "codex_no_session_log"`, and `Message = "codex launched but wrote no session log"`. ## 5. Runner-side try-budget and run-budget exhaustion classification

The runner already distinguishes the two budget kinds via
`loopOut.timedOut && !loopOut.runBudgetExhausted` (try-cap-only) vs
`loopOut.timedOut && loopOut.runBudgetExhausted` (run-budget). Both paths
are updated to carry a non-empty `failure_category`:

- [x] 5.1 **Try-cap-only kill** (`timedOut && !runBudgetExhausted`): after the existing `runBudgetExhausted` reset block at `runner.go:2311-2323` and OUTSIDE the `if failed && !lapPinMismatch && attemptOutcome.CarriesFailureCategory()` block (which is SKIPPED for a try-cap-only kill — `attemptOutcome = OutcomeRunTimeout` at runner.go:2235-2236, and `OutcomeRunTimeout.CarriesFailureCategory()` is false, so `ClassifyError` at runner.go:2254 never runs and the `decision` local is never assigned — do NOT reference `decision` here, it is out of scope). The override fires when `timedOut && !runBudgetExhausted` is true AND no FailureCategory has been set by executor/session/disk-log evidence. The gate condition is: `result.Evidence == nil || (result.Evidence.Category == "")`. If the executor produced evidence with a non-empty Category (e.g. opencode disk log yielded `auth_or_proxy`, codex session log yielded `harness_launch`), the executor's category is authoritative and the override does not fire. When the override fires: set `failureCategory = unidentified_issue`, `failReason = "try budget exhausted; no output"`, and `failureClass = FailureAgent` (the harness did not fail; the runner killed it on the try cap). `CategoryUnidentifiedIssue` maps to `FailureAgent` (does-not-freeze), matching the intent
- [x] 5.2 **Run-budget kill** (`timedOut && runBudgetExhausted`): update the existing block at `runner.go:2311-2323` to set `failureCategory = unidentified_issue` instead of the current `failureCategory = ""` (empty = no signal in telemetry). The `failReason = "run timeout"` and `FailureAgent` class are unchanged. The override gate MUST include `&& failureCategory == ""` (the classifier-produced category from line 2257) — when `ClassifyError` ran and produced a non-empty `failureCategory` (e.g. `agent_error` from a text pattern, or `incomplete_finalization` from dirty-tree), the classifier's category is authoritative. The gate is: `if runBudgetExhausted && failureCategory == "" && (result.Evidence == nil || result.Evidence.Category == "") { failureCategory = unidentified_issue }`. This prevents the override from incorrectly replacing a specific classified failure (e.g. the agent exited with a text-pattern-matched error AND then the run budget was consumed by prior retries)
- [x] 5.3 The `result.Evidence == nil || result.Evidence.Category == ""` gate IS the "real evidence produced" check — if the executor produced Evidence with a non-empty Category, the override does not fire and the executor's category is authoritative. For the try-cap-only kill case, `decision` is out of scope (ClassifyError was never called; see 5.1), so `decision.Evidence` is NOT a valid reference. For the run-budget kill case, the override happens inside the same block where `failureCategory` is currently cleared — the context-dependent gate is essential
- [x] 5.4 The "slow model vs underqualified" diagnostic question is answerable today via the existing try-log tags — do NOT duplicate them into `failure_evidence`. NRQL filters on `outcome IN (run_timeout, handoff_timeout)` plus `timeout_kind` (already emitted at `runner.go:2468` as `try_cap`/`run_budget`/`handoff`), `runtime_ms`, `tool_calls`, `files_changed`, and `last_output_age_ms` (high `tool_calls` + small `last_output_age_ms` = active work in progress; low `tool_calls` + large `last_output_age_ms` = stalled). Document this in the release notes (task 8.3)
- [x] 5.5 Tests: (a) try-cap kill with no evidence → category `unidentified_issue`, fail reason `try budget exhausted; no output`, `FailureClass = agent`; (b) try-cap kill with executor evidence carrying a non-empty Category (e.g. codex session log → `harness_launch`) → existing path, no override, executor's category is authoritative; (c) freeze counter does not increment on a try-cap-kill try; (d) run-budget kill with no evidence and no classifier-produced category → category `unidentified_issue`, fail reason `run timeout`, `FailureClass = agent` (changed from the old empty-category behaviour); (e) run-budget kill where ClassifyError produced a category (e.g. a text-pattern `agent_error`) AND run budget was consumed by prior retries → classifier's category is authoritative (no override); (f) verify no regression in operator-worthy capture for any budget-kill path
